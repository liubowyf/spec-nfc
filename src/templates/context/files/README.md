# `context` 模块使用说明

这个目录存放长期稳定的信息，给人和 AI 共用。

## 文件

- `system.md`：系统定位、依赖、边界、运行约束
- `coding-rules.md`：编码、Git、测试、变更约束
- `architecture.md`：模块划分、分层边界、非功能约束
- `domain.md`：反诈术语、领域对象、关键业务规则
- `update-checklist.md`：上下文维护清单
- `AGENT.md`：给 AI 工具直接使用的上下文约束

## 作为项目长期记忆的职责

本目录承担**仓级长期事实**的首选落点，主要回答：

- 这个仓到底负责什么，不负责什么
- 关键架构边界、领域术语和禁改区是什么
- 哪些编码、测试、协作约束是长期稳定的

如果一条信息会长期影响多个 change / integration，应优先落在本目录，而不是散落在聊天记录里。

## 什么时候更新

- 新增或移除关键模块
- 接口或库表边界发生变化
- 发布约束、合规约束、安全边界发生变化
- 团队的编码和测试规则发生变化
- 领域术语、核心业务规则发生变化

## 维护顺序

1. 先更新 `system.md`
2. 再更新 `architecture.md`
3. 再更新 `domain.md`
4. 最后检查 `coding-rules.md` 是否需要同步

## 使用方式

- 每次大需求启动前先检查 `context/`
- 新人入组时优先阅读本目录
- AI 开始改代码前必须先读本目录
- 如果当前 change 的局部决策与本目录冲突，先回看仓级边界是否需要更新，再决定是否允许变更
